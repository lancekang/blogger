# Blogger Agent Project Instructions

이 파일은 이 저장소의 Codex·Antigravity 공통 에이전트 지침에 대한 단일 원본이다. `.agents/AGENTS.md`는 이 파일을 가리키는 호환 진입점이며, 규칙을 중복 작성하지 않는다.

## 1. 적용 범위와 기본 원칙

- 이 프로젝트는 IT/테크, 패션, 트렌드 분야의 Blogger 포스트 생성과 발행을 지원한다.
- 포스트 생성은 웹앱 내부의 OpenAI API가 아니라 외부 에이전트가 담당한다.
- 비밀 키는 클라이언트 컴포넌트에 노출하거나 하드코딩하지 않고 서버의 `process.env`에서만 사용한다.
- Next.js Dynamic API(`headers()`, `cookies()`, `params`)는 비동기로 처리한다.
- Blogger 연동은 `lib/blogger.ts`의 fetch 기반 REST 방식을 유지한다.

## 2. 최신 정보 검증

- 현재 날짜와 연도는 실행 환경에서 확인한다. 특정 연도를 영구적으로 고정하지 않는다.
- 글을 작성하기 전에 현재 에이전트 환경에서 제공되는 웹 검색 또는 브라우저 기능으로 최신 정보를 확인한다. 특정 도구명에 의존하지 않는다.
- 제품 스펙은 제조사 공식 문서와 발표 자료를 우선하고, 필요할 때 신뢰할 수 있는 복수 출처로 교차 검증한다.
- 가격은 정가뿐 아니라 다나와와 국내 오픈마켓의 조회 시점 실구매가 범위를 확인한다.
- 확인하지 못한 정보는 추정하지 말고 원고에 불확실성을 명시하거나 제외한다.

## 3. 포스트 품질과 형식

- 포스트 본문은 최소 3,000자 이상의 매거진 스타일로 작성한다.
- 단순 나열 대신 용도별 비교와 선택 기준을 제공한다.
- 라벨은 중복 없이 핵심 키워드 3~5개만 사용한다.
- 다나와 검색 링크는 불완전한 퍼센트 인코딩 대신 `keyword=샤오미+패드+8+프로`와 같은 안전한 쿼리를 사용한다.
- 쇼핑 또는 가격 비교 링크 아래에는 다음 문구를 11~12px 이탤릭체로 표시한다: `※ 실시간 가격 비교 링크는 시장 상황 및 조회 시점에 따라 실제 판매 가격과 재고가 상이할 수 있습니다.`
- 앱과 소프트웨어는 공식 홈페이지 또는 검증된 App Store·Google Play 고유 링크를 사용한다.
- 실재하는 특정 제품(IT 기기, 마우스 등)의 실제 외형·디자인과 다른 이미지가 생성될 가능성이 높거나 정확한 묘사가 어려울 경우, 해당 콘텐츠에는 이미지를 제외하고 텍스트와 구조적 레이아웃 위주의 깔끔한 구성으로 포스트를 생성한다.
- 포스트 본문 하단의 주의사항 및 필수 체크 영역은 단순 리스트(`<li>`) 대신, 항목별 이모지 아이콘, 주색상 매칭 왼쪽 4px 테두리 바, 제목과 설명글이 명확하게 구분되는 프리미엄 카드형 레이아웃을 사용한다.

## 4. `블로거` 명령과 자동화

사용자가 `블로거`를 독립 명령으로 입력하거나 큐 자동화를 명시적으로 요청했을 때만 아래 파이프라인을 실행한다. 일반 문장에 포함된 단어만으로 자동화를 만들지 않는다.

1. `http://localhost:3000` 상태를 확인한다. 필요하고 현재 실행 환경이 지속적인 백그라운드 프로세스를 지원할 때만 `npm run dev`를 실행한다.
2. 자동화 기능이 제공되는 환경에서는 5분 주기(`*/5 * * * *`)의 프로젝트 자동화를 구성한다. 특정 스케줄 도구 이름이나 호출 형식을 가정하지 않는다.
3. Codex App에서는 반드시 현재 로컬 프로젝트에서 실행한다. 큐가 로컬 파일에 있으므로 별도 worktree를 사용하지 않는다.
4. Codex App과 로컬 PC가 실행 중이어야 자동화가 동작한다는 점을 사용자에게 알린다.
5. 현재 환경에서 자동화 생성 기능을 사용할 수 없으면 등록했다고 주장하지 말고, 지원되는 설정 절차를 사용자에게 안내한다.
6. 자동화 등록 여부와 관계없이 `POST /api/ai/request-post/claim`으로 요청을 원자적으로 선점하고, 응답에 요청이 있을 때만 즉시 처리한다. `requestId`를 지정하지 않으면 가장 오래된 처리 가능 요청을 가져온다.

자동화 프롬프트에는 다음 의도를 포함한다: `$blogger-helper를 사용해 현재 로컬 프로젝트의 대기 큐를 확인하고 pending 요청이 있으면 사실 검증 후 포스트를 생성해 저장한다. 대기 요청이 없으면 파일을 변경하지 않는다.`

## 5. 큐 결과 저장

- 작업 시작 전 `POST http://localhost:3000/api/ai/request-post/claim`에 `{ workerId }`를 보내 요청을 선점한다. 응답의 `request`가 `null`이면 처리할 작업이 없는 것이다.
- 결과는 우선 `POST http://localhost:3000/api/ai/pending-posts`에 `{ title, content, labels, images, requestId }` JSON 객체로 저장한다.
- `requestId`를 반드시 전달하고 API가 성공 응답을 반환한 뒤에만 요청이 완료됐다고 판단한다.
- `pending-posts/*.json`을 문자열 템플릿이나 패치로 직접 조립하지 않는다.
- API를 사용할 수 없어 파일 저장이 꼭 필요하면 `JSON.stringify` 또는 `json.dump(..., ensure_ascii=False)`를 사용하고 임시 파일 작성 후 rename하는 원자적 교체 방식으로 복구용 초안만 저장한다. 이 직접 저장은 완료 처리가 아니며, 서버가 복구되면 동일한 `requestId`와 원고를 결과 API에 다시 전송해 큐 상태까지 완료해야 한다. 요청 JSON의 상태를 직접 변경하지 않는다.
- 실패 시 `PATCH http://localhost:3000/api/ai/request-post`에 `{ id: requestId, action: "fail", error }`를 보내 `failed` 상태와 원인을 기록한다. 완료 처리하지 않으며 재시도 필요 여부를 보고한다.
- 에이전트 발행 모드가 `Blogger 초안으로 자동 저장` 또는 `Blogger에 즉시 자동 발행`으로 설정된 경우, 백그라운드에서 자동 업로드가 수행된다. 이때 Blogger API의 429 Too Many Requests (Rate Limit) 에러를 예방하기 위해, 포스트 등록 성공 시마다 최소 4초간의 대기 지연(Cooldown)을 반드시 준수하여 요청한다.

## 6. 검증과 보고

- 코드 변경 후 영향 범위에 맞게 최소 `npm run lint`와 `npm run build`를 실행한다.
- 사용자에게 서버 상태, 자동화 등록 여부와 실행 위치, 처리한 요청 ID, 실패 항목을 구분해 보고한다.
- 실제로 실행하거나 확인하지 않은 작업을 완료됐다고 표현하지 않는다.
